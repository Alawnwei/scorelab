# 杯赛全流程修复 + datafc 激活 + 预测来源标识

> 今日改动覆盖三大块：杯赛预测记录补全、俱乐部数据源切换到 Sofascore、预测来源标识体系建立。

---

## 一、杯赛预测记录补全

### 1. predictions_db 写入（`predict.py`）

**问题：** `predict.py` 运行后只写 MD 文件，不写 `predictions_db.json`，导致杯赛预测在数据库里没记录。

**修复：** `predict.py` 末尾直写 JSON（避免同进程 stdout 冲突），内联温度缩放 T=1.60，查重保护。

```python
# predict.py 新增 ~30 行
if final_rating in WRITE_RATINGS:
    # 直接读写 predictions_db.json（注入式，不 import prediction_db）
    # 应用温度缩放 → 查重 → 写入
```

**影响：** `python skill/predict.py --home "X" --away "Y" --league wm26` 现在同时产出 MD + predictions_db 记录。

### 2. odds-api.io 回退写 PnL（`datafc_provider.py`）

**问题：** `auto_fill_odds()` 的 datafc 分支有 `add_record()` 写 PnL，但 odds-api.io 回退分支（杯赛必经路径）漏了。

**修复：** odds-api.io 分支补上 `add_record()`，`odds_only=True` 时跳过（与 datafc 分支一致）。

**影响：** 杯赛预测现在写 `pnl_records.json`。

### 3. 国家队映射补全（`datafc_provider.py`）

| 缺什么 | 补了什么 |
|:-------|:---------|
| TEAM_EN | 新增 23 支国家队英文名（乌兹别克斯坦→Uzbekistan, 科特迪瓦→Ivory Coast, 伊朗→Iran 等） |
| TEAM_DE | 新增 "乌兹别克斯坦" → Usbekistan（原已有"乌兹别克"） |

## 二、datafc (Sofascore) 激活

### 4. DATAFIC_ALIVE 切换

**问题：** 旧设定 `DATAFIC_ALIVE=False`，俱乐部预测使用 OpenLigaDB 但无数据，λ 退化为联赛均值（两队相同），七维评分退化为默认值 5.0。

**改动：** 验证 Sofascore 可达后设为 `True`。

**效果对比（索尔纳 vs 哥德堡 - 瑞典超）：**

| 指标 | 改前 (OpenLigaDB) | 改后 (Sofascore) |
|:-----|:-----------------:|:----------------:|
| 数据源 | openligadb_no_data | datafc_goals_fallback |
| 索尔纳 xG_for | 1.30（联赛均值） | **1.40**（5场实际） |
| 哥德堡 xG_for | 1.30（联赛均值） | **2.00**（5场实际） |
| 索尔纳进攻评分 | 5.0（默认） | **4.7** |
| 哥德堡进攻评分 | 5.0（默认） | **5.3** |
| P_base_raw | 0.5（纯平局） | **0.52**（有倾向） |

### 5. 轮次动态检测 + 已赛过滤

**问题 1：** `get_team_recent_matches()` 固定扫描 20→1 轮，但瑞典超有 30 轮、K1 有 33 轮 → 最新比赛没扫到。

**修复：** 用 `season_rounds_data()` 获取实际最大轮次，失败时默认 38 轮。

**问题 2：** 扫描到最新轮次（如 28-30 轮）但状态全是 "Not started"（未来比赛），`status == "Ended"` 后置过滤把所有比赛都筛掉了 → 返回 0 条。

**修复：** 在扫描循环内就过滤 `status == "Ended"`，非完赛的比赛不被收集。

### 6. 队名映射修正（`datafc_provider.py` TEAM_EN）

Sofascore 队名与旧 TEAM_EN 英文名不一致，导致 `_find_team_in_matches` 匹配失败：

| 球队 | 旧 TEAM_EN | Sofascore 队名 | 改后 |
|:-----|:----------|:---------------|:-----|
| 哥德堡 | IFK Gothenburg | IFK Göteborg | IFK Göteborg |
| 蔚山现代 | Ulsan Hyundai | Ulsan HD | Ulsan HD |
| 大田市民 | Daejeon Citizen | Daejeon Hana Citizen | Daejeon Hana Citizen |
| 厄格里特 | ❌ 缺失 | Örgryte IS | 新增 Orgryte IS |
| 安养FC | ❌ 缺失 | FC Anyang | 新增 FC Anyang |
| 富川FC | ❌ 缺失 | Bucheon FC 1995 | 新增 Bucheon FC 1995 |

### 7. 杯赛跳过 Sofascore 直接走 OpenLigaDB

**问题：** `DATAFIC_ALIVE=True` 后杯赛（wm26）也尝试连 Sofascore，但杯赛数据从中国访问慢，导致超时。

**修复：** `get_team_recent_matches()` 中 `type=world_cup/knockout/uefa` 直接返回空，触发 OpenLigaDB 回退。

### 8. 磁盘双层缓存

**问题：** 每次新进程冷启动需要 2-3 分钟（扫描 30+ 轮次的 Sofascore API 调用）。

**修复：** 新增两个持久化缓存文件：

| 缓存文件 | 缓存内容 | 大小参考 |
|:---------|:---------|:---------|
| `recent_matches_cache.json` | `get_team_recent_matches` 返回的 DataFrame | ~2.5KB/队 |
| `team_strength_cache.json` | `estimate_team_strength` 返回的 xG 字典 | ~500B/队 |

首次预测~2分钟，后续**<1秒**。

### 9. NaN 传播防护

**根因：** 脏缓存中 `xG_for=nan` → `adjust_lambda(nan, None, None)` → λ=6.00（cap）→ NB 畸变 → 客胜 79% → +912% EV（虚假信号）

**修复：** 三层 NaN/None 防护：

| 层 | 位置 | 行为 |
|:---|:------|:------|
| ① | `predict.py:adjust_lambda()` | NaN/None → 1.0（无调整） |
| ② | `datafc_provider.py:estimate_team_strength()` | NaN xG → 联赛均值 |
| ③ | `datafc_provider.py:缓存写入` | NaN 或 matches=0 不写入 |

## 三、预测来源标识体系

### 10. prediction_db.py --source 参数

**问题：** `predictions_db` 中所有记录 `source=manual` 或 `md_file`，分不清是人工分析还是自动管线产出。

**改动：**

```python
# prediction_db.py
add_prediction(..., source="manual")  # 新增参数
# CLI: --source {manual, auto_pipeline, predict.py}
```

**来源标识规范：**

| source 值 | 写入途径 | 含义 |
|:----------|:---------|:-----|
| `manual` | `prediction_db.py --add` | 人工分析后添加 |
| `auto_pipeline` | `run_predict.sh --add` | 自动脚本简化管线 |
| `predict.py` | `python skill/predict.py` | 系统1完整引擎 |
| `md_file` | `--import-md` | 历史导入 |
| `pnl_tracker` | `--import-pnl` | PnL 导入 |

## 五、赔率定时刷新 + CLV 追踪

### 17. 带时间戳的 odds 文件保存（`datafc_provider.py`）

**问题：** `odds_*.json` 文件只写入一次，CLV 需要多版本比较，但只有单版本。

**修复：** 新增 `_save_odds_file()` 函数，在 `auto_fill_odds()` 的两条路径（datafc + odds-api.io）中每次获取赔率时都保存带时间戳的新版本文件。

```python
# 文件名: odds_主队_客队_YYYYMMDD_HHMMSS.json
# 内容: 1x2 赔率 + 全市场数据（大小球、BTTS、角球等）
```

### 18. 赔率刷新命令（`prediction_db.py`）

**新增 `--refresh-odds` 命令：** 扫描所有 `pending` 状态的预测，重新调用 `auto_fill_odds()` 获取最新赔率，每次生成新版本 odds 文件。

```bash
python skill/prediction_db.py --refresh-odds
```

### 19. 全能监控模式（`prediction_db.py --watch`）

`--watch` 模式从"纯结果回填"升级为"结果+赔率+校准"三合一：

```
while True (每60秒):
  1. sync_results()              ← 从 auto_results.json 回填比分
  2. sync_from_datafc("wm26")    ← 从数据源同步完赛结果
  3. 有新结果？→ 自动 calibration  ← 新增
  4. refresh_odds()              ← 刷新待赛赔率  ← 新增
```

`run_predict.sh --watch` 改为调用 `prediction_db.py --watch`（不再用 shell 简化版）。

## 六、温度缩放 T=1.60

### 20. predict.py 主管线应用温度缩放

**问题：** 校准最优 T=1.60 算出后只在 `add_prediction()` 入库时应用，`predict.py` 输出的 P_final 仍是 T=1.0（未校准），强队概率系统性虚高 10-15%。

**修复：** `predict.py` 第 261 行，在 P_final 计算后立即应用温度缩放：

```python
TEMP_T = 1.60
_pt = p_final ** (1.0 / TEMP_T)
_qt = (1.0 - p_final) ** (1.0 / TEMP_T)
p_final = _pt / (_pt + _qt)
```

**效果示例：**

| 原始 P_final | 缩放后 | Δ |
|:-----------:|:------:|:-:|
| 0.80 | 0.704 | -9.6pp |
| 0.61 | 0.571 | -3.9pp |
| 0.50 | 0.500 | 0.0pp |

同时修正 predictions_db 写入处的重复缩放（防双吃）。

## 七、统一启动入口

### 21. run_predict.sh 重构

**问题：** `run_predict.sh --home` 走的是独立 shell 路径（`xg_estimator.py` + shell 数学算 P_final），与 `predict.py`（完整系统1引擎）完全脱节。

**修复：** `run_prediction()` 改为直接调用 `predict.py`，所有参数透传。

**新增命令：**

| 命令 | 功能 |
|:-----|:------|
| `bash run_predict.sh --home X --away Y --league Z` | 单命令预测（完整引擎） |
| `bash run_predict.sh --all X Y Z` | 预测 + 进入监控 |
| `bash run_predict.sh --watch` | 全能监控（结果+赔率+校准） |
| `bash run_predict.sh --status` | 仪表盘 |

### 22. run_predict.bat（已删除）

因 Git Bash (MSYS2) 系统性破坏批处理文件的 `%1` 参数语法，`.bat` 无法从 Git Bash 正常运行。已删除，用户统一用 `.sh` 版本。

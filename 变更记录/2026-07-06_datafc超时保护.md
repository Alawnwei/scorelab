# datafc API调用超时保护 — 变更说明

> 修复datafc（SofaScore）API调用无超时导致的脚本卡死问题。

---

## 问题

`datafc_provider.py` 中所有网络请求没有 timeout 参数。当 SofaScore API 不可达时，`predict.py` 无限阻塞，不输出任何信息，需要手动 kill 进程。

受影响函数：
- `get_matches()`, `estimate_team_strength()`, `get_odds()`
- `get_formations()`, `get_lineups()`, `get_pregame_form()`, `get_match_details()`
- `get_team_elo()`, `infer_m_factors()`

## 修改

### 新增函数

`_call_with_timeout(fn, timeout=12)` — 基于 `concurrent.futures.ThreadPoolExecutor` 的通用超时包装器。

```python
def _call_with_timeout(fn, timeout=12):
    from concurrent.futures import ThreadPoolExecutor, TimeoutError as _FETimeoutError
    with ThreadPoolExecutor(max_workers=1) as _pool:
        _future = _pool.submit(fn)
        try:
            return _future.result(timeout=timeout)
        except _FETimeoutError:
            raise Exception(f"datafc 调用超时 ({timeout}s)")
```

### 修改位置

| 步骤 | 行号 | 包裹的调用 | 超时 |
|:---:|:----:|:-----------|:----:|
| ① 基本面 | 631-632 | `get_matches()` ×2 | 12s |
| ② xG数据 | 658-660 | `estimate_team_strength()` ×2 | 12s |
| ③ 赔率 | 729-731 | `get_matches()` + `get_odds()` | 12s |
| ④ 情报 | 897-914 | `get_matches()` + 4个情报调用 | 12s |
| ⑤ M因子 | 945 | `infer_m_factors()` ×2 | 15s |
| ELO | 1005-1006 | `get_team_elo()` ×2 | 12s |

### 超时后的行为

超时抛 `Exception("datafc 调用超时 (Xs)")` → 被各步骤已有的 `except Exception as e:` 捕获 → 自动走 OpenLigaDB 回退路径或跳过该步骤。

| 慢但未超时（1s sleep + 3s timeout → 正常返回） | ✅ |
| 语法检查 | ✅ |

## 回滚方式

- 删除 `_call_with_timeout` 函数定义
- 将所有 `_call_with_timeout(lambda: ...)` 恢复为直接调用
